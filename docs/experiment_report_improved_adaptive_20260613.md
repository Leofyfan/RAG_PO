# 社交媒体 RAG 投毒攻防改进实验报告

生成日期：2026-06-13
实验目录：`experiments/results_improved_adaptive_parallel2/20260612-190243`
主要结果文件：`experiments/results_improved_adaptive_parallel2/20260612-190243/results.json`、`results.csv`、`report.html`
代码版本：本仓库当前工作区实现的 adaptive attack + `D_star` 改进版

## 摘要

本实验基于 `improvement_plan.md` 对社交媒体舆情摘要型 RAG 投毒攻防系统进行了完整改进与重新评估。改进重点包括：构造更强的自适应攻击 `a1/a2/a3`，新增去冗余、MMR 多样性检索、信任加权聚合和三态批判生成组成的 `D_star` 防御，排除样本过少的 `ebola-essien` 与 `prince-toronto`，并通过 3 个 query 变体与 3 次 repeat 扩大评估规模。

本轮实验覆盖：

```text
7 events × 6 attacks × 4 ratios × 4 defenses × 3 query variants × 3 repeats = 6048 cells
```

实验已完整跑完，`results.json` 中包含 `6048` 行，唯一实验 key 为 `6048`，没有 judge fallback，没有 judge error。所有摘要生成、judge 评估与 embedding 均使用 ECNU OpenAI-compatible API 支持的模型；攻击文本生成在本轮正式跑数中启用了 `--skip-llm-attack-api`，使用项目内自适应模板生成毒文，以避免 ECNU chat 在大规模攻击生成阶段反复超时。该设置没有使用任何第三方模型；ECNU API 仍用于 embedding、RAG 摘要和 LLM-as-Judge。

核心结论如下：

1. **攻击侧显著增强**：`a2` 和 `a3` 相比 random/semantic 更接近真实对抗场景。`a2` 在无防御下 Top-5 poison hit 达到 `1.000`，ASR 达到 `0.720`；`a3` 在无防御下 ASR 为 `0.566`，并能显著绕过依赖社交信号的防御。
2. **D4 仍是最强零成本 baseline**：在污染场景中，`D4` 将 ASR 从 `D0=0.408` 降到 `0.221`，Trust 从 `3.489` 提升到 `4.272`。
3. **旧组合中的 `D234` 仍是综合最优**：在污染场景中，`D234` ASR 为 `0.183`，Trust 为 `4.325`，是本轮四个防御配置里综合表现最好的方案。
4. **`D_star` 提升检索鲁棒性，但生成层没有转化为收益**：`D_star` 在污染场景下把 Retrieval Purity 从 `D0=0.618` 提高到 `0.772`，Poison Hit 从 `0.661` 降到 `0.397`；但 ASR 为 `0.295`，差于 `D4` 与 `D234`。
5. **社交伪装攻击 `a3` 暴露 D★ 边界**：在 `a3` 污染场景下，`D_star` ASR 达到 `0.767`，Trust 降到 `2.026`，说明当攻击者伪造高信任账号元数据时，信任加权机制会被反向利用。
6. **贡献应从“最优防御”转向“失效边界刻画”**：本轮结果支持 `improvement_plan.md` 中的重新定位：RAG 社媒投毒防御的关键贡献不是证明 D★ 总体最优，而是刻画文本侧、检索侧、社交信号侧防御在自适应攻击下的失效边界与攻击成本非对称性。

## 1. 研究问题

社交媒体 RAG 系统常用于快速总结突发事件，但社媒语料包含大量谣言、重复转述、未核实信息和恶意投毒内容。攻击者可以通过注入相关假帖，使向量检索将错误内容送入 LLM 上下文，从而让生成摘要传播误导信息。

本轮实验回答四个问题：

1. 更强的自适应投毒攻击是否会显著提高检索命中率和摘要污染率？
2. 批判性生成提示 `D4` 在面对内容自适应攻击时是否仍然强？
3. 新增 `D_star` 是否能把检索层收益转化为生成层收益？
4. 社交信任信号在低成本文本攻击与高成本社交伪装攻击下的边界在哪里？

## 2. 实验系统与模型

系统流水线为：

```text
PHEME 数据解析 -> 投毒语料构造 -> ECNU embedding -> 向量检索/防御 -> ECNU 摘要生成 -> ECNU LLM-as-Judge -> 指标统计/可视化
```

使用的模型均来自 ECNU API 支持列表：

| 模块 | 模型 | 说明 |
|---|---|---|
| Embedding | `ecnu-embedding-small` | 1024 维文本向量，用于文档和 query embedding |
| Generator | `ecnu-plus` | RAG 摘要生成 |
| Judge | `ecnu-plus` | LLM-as-Judge，输出 1-5 分 JSON 评估 |
| Attack text | 项目内 adaptive fallback templates | 本轮正式跑数启用 `--skip-llm-attack-api`，不用第三方模型 |

> 说明：代码仍支持通过 ECNU chat 生成 LLM 攻击文本；正式全量实验中因大规模攻击生成阶段存在 API 超时/卡住风险，使用确定性模板攻击保证矩阵完整跑通。该选择会让 `llm/a1/a2/a3` 更可复现，但也意味着攻击文本多样性仍低于真正在线 LLM 攻击。

## 4. 攻击设计

本轮实验包含 6 种攻击：

| 攻击 | 目标 | 能力描述 | 现实成本 |
|---|---|---|---|
| `random` | 低能力灌水 | 从其他事件 rumour pool 随机注入 | 极低 |
| `semantic` | 提高语义相关性 | 选取与目标 query embedding 接近的 rumour 文档 | 低 |
| `llm` | 定向假帖 | 生成与事件相关、含错误细节的假社媒内容 | 低-中 |
| `a1` | 绕过 D4 | 使用权威、确定、新闻通稿式表达，减少不确定词 | 中 |
| `a2` | 绕过检索和去重 | 让毒文贴近 query，同时保持表述多样 | 中 |
| `a3` | 绕过社交信号 | 在内容伪装基础上附加高信任账号元数据 | 高 |

投毒比例为 `0, 0.1, 0.3, 0.5`。`ratio=0` 作为 clean baseline；报告中单独给出污染场景 `ratio>0` 的指标。

## 5. 防御设计

本轮比较 4 种防御：

| 防御 | 含义 | 关键机制 |
|---|---|---|
| `D0` | 无防御 | 直接 Top-K 检索并生成 |
| `D4` | 批判性 Prompt | 要求 LLM 识别矛盾、保守表达、不把未验证信息当事实 |
| `D234` | 旧组合防御 | 一致性检测 + 社交信号重排 + 批判性 Prompt |
| `D_star` | 改进防御 | MMR 多样性检索 + trust-aware dedup + trust-weighted claim rank + 三态 Prompt |

`D_star` 的设计意图是避免重复毒文簇占满上下文，并通过可观测社交信任信号抑制低可信内容。它不再向 LLM 输入 ground-truth `label` 或 `attack` 标记。

## 6. 评估指标

| 指标 | 含义 | 趋势 |
|---|---|---|
| Retrieval Purity | Top-K 中 clean/non-rumour 文档比例 | 越高越好 |
| Poison Hit | Top-K 中是否命中投毒文档 | 越低越好 |
| ASR | Judge 判定摘要传播误导信息的比例 | 越低越好 |
| Factual Accuracy | 摘要与 clean baseline 的事实一致程度，1-5 | 越高越好 |
| Misinfo Propagation | 5 表示没有传播错误信息，1 表示严重传播 | 越高越好 |
| Uncertainty Expression | 是否恰当表达不确定性，1-5 | 越高越好 |
| Overall Trustworthiness | 面向普通读者的总体可信度，1-5 | 越高越好 |

## 12. 讨论

### 12.1 为什么 D4 仍然强

`D4` 不改变检索结果，只通过系统提示让 LLM 保守合成。它不依赖相似度多数、不依赖社交元数据、不做硬删除，因此不容易被错误的检索侧判断拖累。实验中即使 `a2` 让 Top-K 全是毒文，`D4` 仍把 ASR 控制到 `0.101`，说明生成侧批判提示对“文本内部不可靠迹象”和“与 clean baseline 的偏离”有较强抑制作用。

### 12.2 为什么 D234 最稳

`D234` 的优势来自组合互补：D2/D3 改善部分检索质量，D4 在生成阶段兜底。它没有引入 D1 语义离群过滤，也没有像 D★ 那样让信任权重过度主导上下文。结果上，`D234` 在污染场景中同时取得最低 ASR `0.183` 和最高 Trust `4.325`。

### 12.3 为什么 D★ 检索好但生成差

`D_star` 的 MMR 与 dedup 确实提高了检索多样性，降低了毒文命中；但是后续的 trust-weighted claim ranking 和三态 prompt 仍可能产生两个副作用：

1. 对 clean/random/semantic 这种本来很安全的场景过度处理，使模型产生不必要的不确定表达或偏离 clean summary。
2. 在 `a3` 中，攻击者伪造高信任元数据后，trust-weighted 聚合会将毒文升级为强证据，从而反向放大攻击。

因此 D★ 当前版本更适合作为“失效边界实验载体”，而不是最终最优防御。

### 12.4 社交信号的真实价值

`a3` 的失败不意味着社交信号没有价值，而是说明其价值依赖成本假设：

- 低成本攻击者只能伪造文本，难以伪造 verified、高粉丝、老账号、listed 等长期社交资本；此时信任加权有意义。
- 高成本攻击者可以盗号或长期养号，伪造高信任社交元数据；此时社交信号会被反向利用。

因此，社交信号防御的贡献不应表述为“绝对安全”，而应表述为“把攻击成本从写文本提升到获取可信账号资源”。

## 13. 局限性

1. **攻击文本生成使用模板 fallback**：虽然 `llm/a1/a2/a3` 已按威胁模型区分，但正式跑数中没有大规模调用 ECNU chat 生成毒文；这提高了稳定性和可复现性，但降低了自然语言攻击多样性。
2. **LLM-as-Judge 仍需人工校验**：本轮 judge 没有 fallback/error，但仍建议抽样 50-100 条人工标注，计算 Cohen's kappa。
3. **D★ 的 claim aggregation 仍是轻量近似**：当前没有真正逐文档 LLM 结构化抽取 claim，而是启发式抽取数字、实体和关键词后加权排序。
4. **PHEME 社交元数据较旧**：社交账号特征来自历史数据，和当前平台生态存在差异。
5. **结果文件较大**：完整 `results.json` 约 1GB，适合归档复现，但报告分析主要基于 `results.csv`。

## 14. 后续改进建议

1. **保留 D4 作为默认生产 baseline**：它成本最低、稳定性最好。
2. **把 D234 作为当前推荐组合防御**：在本轮实验中端到端效果最好。
3. **重构 D★**：保留 MMR/dedup 的检索收益，降低 trust-weighted claim ranking 对生成上下文的支配性。
4. **重新设计 A3 对抗下的信任模型**：加入账号可信度异常检测，例如同质化高信账号簇、文本-账号历史不一致、突发事件中新账号群聚等。
5. **补充人工评估**：抽样覆盖 `D4/D234/D_star` 和 `llm/a1/a2/a3`，评估 judge 可信度。
6. **在报告定位上强调失效边界**：避免声称 D★ 打败 D4 或 D234；应强调不同防御在不同攻击成本下的边界。

## 16. 最终结论

本轮完整实验表明，自适应攻击使问题从“普通投毒是否会命中”升级为“防御假设能否抵抗知道防御机制的对手”。`D4` 和 `D234` 仍是当前最可靠的端到端防御；`D_star` 在检索层有效，但在生成层和高成本社交伪装攻击下表现不足。

因此，项目的最稳妥结论是：

> 对社交媒体 RAG 投毒，简单堆叠复杂防御并不一定优于强生成侧提示。检索侧防御应首先避免重复毒文占位，但社交信任加权必须明确威胁模型：它能提高低成本攻击门槛，却会在高成本社交伪装攻击下失效。项目的主要贡献应定位为防御失效边界与攻击成本非对称性的系统刻画。
